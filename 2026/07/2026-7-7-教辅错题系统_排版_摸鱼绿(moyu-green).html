我在做一个给教辅机构用的小产品。

起因是一个很具体的麻烦。

机构用企业微信建了家长群。每次下课，老师就在群里 @家长，发一堆孩子这周的作业、错题照片，再配一句：

三年级数学第3讲，错题5张。

家长当时点开看一眼，过了就过了。

到了期末，问题全冒出来。

一个家长问：老师，我家孩子这学期到底哪些题老错？老师翻聊天记录翻了俩小时。

机构老板也问：哪个年级哪个知识点错得最多，我好排复习课？没人答得上来。

几千条群聊消息，就这么散在那里。

信息进去了，但谁也掏不出来。

我盯着这个场景想了一阵，意识到一件事：这不是“加个人去整理聊天记录”能解决的。消息每天都在产生，人根本整不过来。

得让机器自动干。

PRINCIPLE · 不改习惯

老师已经习惯了在群里发。

所以我把所有力气都花在后端：老师照常在群里发，我在背后把会话自动沉淀成结构化的档案。

第一层：把企业微信群聊存下来

LAYER 01 · 会话存档

先说最反直觉的一件事。

企业微信的消息，不是你写个接口、它主动推给你那么简单。

普通的“接收消息”回调，只能接到用户发给应用本身的那几条。家长群里老师、家长互相发的消息，回调根本收不到。

想拿到完整的群聊记录，得用企业微信另一个能力，叫会话内容存档。

而这个能力，官方给的方式很硬核：它不给你 HTTP 接口，给你一个 C 语言的动态库。

我的后端是 Node，用的 NestJS。Node 直接调不了 C 库。

！第一个坑 🕳

官方文档默认你会用 C 或者 Java，Node 的例子几乎没有。

我的办法是：自己拿 C 写了一个小包装程序，把“拉取一批消息”这个动作包成一条命令行。Node 这边用子进程去调它，拿回一段 JSON，再接着处理。

调通之后，还有一道加密关。

会话存档拿到的每条消息，是双重加密的：

一个叫 encrypt_random_key 的字段，是企业微信用公钥加密了一段随机密钥。你得拿你自己存着的 RSA 私钥 去解它，解出来才是真正的随机密钥。

然后用这个随机密钥，再去解真正的消息体 encrypt_chat_msg。

听起来绕，其实就是两把锁套着。私钥文件得在服务器上放好，丢了或换了，就对不上了。

再有一个细节：拉消息是按一个叫 seq 的序号续传的，不是按时间。

每拉一批，记住下一个 seq，下次从这儿接着拉。我把这个状态写进一个小文件，服务重启了也能接着跑，不重复也不漏。

最后加个定时器，每十分钟自动拉一次今天的消息。

这一层忙活完，消息终于稳定地落进了我自己的数据库。

但到这一步，它还只是一堆带图片的消息，没用。

第二层：让 AI 认出“这是谁的什么作业”

LAYER 02 · 大模型分类

这才是真正解决问题的一层。

我手里现在有一堆消息：一张作业照片，前面或后面跟着老师一句“三年级数学第3讲 @小明妈妈 错题5张”。

我需要把这句话变成五个字段：年级、家长（学生）、学科、讲次、图片张数。

这样照片才能挂到对的人、对的课下面。

这事用正则，写不了一点。

因为老师写字极其随意：错别字、漏标点、@ 丢了、语序乱，有时候“第3讲”写成“第3课”，有时候写成“第三讲”。

你写死规则，第二天就被新说法打脸。

这里就是大模型该上的地方。我接了 DeepSeek。

设计上有几个关键决定，我挨个说。

第一，用 JSON 模式，锁死输出格式。

我不让模型自由发挥。我让它只能返回一个 JSON，固定五个字段：年级、家长、学科、讲次、图片张数。多了不要。

这样我后面写库、查询、聚合，全是确定的结构，不用再去解析自然语言。

第二，prompt 分两段，一段英文讲规则，一段中文补约束。

这个组合是我试出来的。

英文那段告诉它：你是要从一句中文消息里，提取作业照片的元数据。老师可能写得很随意，你要语义理解，不要死抠模板。

中文那段补几条硬约束：@ 要去掉、讲次统一成“第 NN 讲”、抽不出就返回空。

第三，temperature 调到 0。

分类这种活，不需要创造力。我要的是稳定、可复现。同一个输入，跑十次最好十次一样。

第四，也是最要紧的一条：宁可漏，不要编。

我在 prompt 里反复强调——文本里没有的学生、学科、讲次，不许编。图片张数判断不出来，就返回 0，这条消息直接跳过，不做归类。

为什么这么较真？

因为这是教育数据。模型要是把张三的错题挂到李四头上，比不归类严重得多。漏了顶多人工补一下，错了就是家长看到别人家孩子的错题，信任直接崩。

这一层跑完，“一张照片 + 一句随手的文字”，终于变成了——

一张照片 + 一句话，变成

三年级 / 数学 / 第 3 讲 / 小明妈妈 / 5 张

照片开始有了归属。

第三层：图片上云 + 后台看得见

LAYER 03 · 上云可视化

最后一步，相对常规。

图片传到阿里云的对象存储（OSS）。数据库里只存路径，不存图本身——库轻、查询快、也便宜。

然后写了个后台。老师、家长登录进去，按年级、按学生、按讲次看。

点开某个孩子，这学期所有错题按时间排好，哪一讲错了多少、错的什么，一屏看完。

这一层的价值不在技术，在“看得见”。

前面两层干了那么多活，就是为了让最后这一屏能出现。

我把这一屏给机构的人看，对方说了一句：

机构的人看了一眼，说

这个，我能拿去给来咨询的家长看。

——这就是我接下来想做的事的起点。

PART


接下来：让 AI 看懂照片，自己出错题

NEXT · 多模态 + 出题

写到这儿，必须说一句实话。

现在的 AI，只分类了文字。它还没看照片本身。

它知道“这是小明妈妈的三年级数学第3讲”，但它不知道照片里到底哪道题错了、错在哪、考的是什么知识点。

这是我接下来要做的，也是这个产品真正想变成的样子。

下一步分三步。

STEP 01

让 AI 看懂照片内容

接多模态模型，或者先走 OCR。把一张错题照片变成结构化的内容：哪道题、什么题型、考哪个知识点、孩子错在哪一步。

这件事现在的大模型已经能做得不错。难点不在能不能认，在认得稳不稳、成本压不压得下来。

STEP 02

搭两个维度

这是我整件事里最看重的设计。


年级学科维度

把全机构的错题按年级、按学科、按知识点聚合。一看就知道——初二数学的“一次函数”这个知识点，全年级错了一半人。老板拿这个排复习课，老师拿这个调教学重点。


学生维度

单独看一个孩子。他这学期哪些知识点反复错、错题是不是在收敛——同一个知识点，第三次还错，还是已经对了。

家长最想看的就是这个。不用老师总结，他自己就知道孩子卡在哪。

STEP 03

让 AI 出对应的题

知道了薄弱点，就让模型按这个知识点、这个难度，出几道同类题，给孩子针对性练。

错的地方反复练，对的地方不再浪费时间。

这一步做完，整个东西就从——

从“错题档案”

变成“错题驱动的针对性训练”

对一个教辅机构来说，这才是真正能当卖点的：我们不光上课，我们能看到每个孩子的薄弱点，还能针对它出题。

而对方那句“能拿去给家长看”，到这里就升级成了一套完整的招生素材——进度可视化、薄弱点可量化、训练可追踪。

这些东西，全是从最开头那些散在群聊里的照片里，自动长出来的。

EPILOGUE · 写在最后

我以前是写页面的。

把设计稿变成网页，做得再漂亮，那也是别人定好的东西。

这一个项目不一样。它是从一个真实生意的真实麻烦里长出来的——机构老板的抱怨、老师的习惯、家长的需求，每一层都不是我拍脑袋想出来的。

然后我得一层一层解决：会话存档的 C 库怎么调、双重加密怎么解、大模型怎么用才不会乱编、图片怎么存才划算、哪一步该自动化、哪一步该留给人。

这些，没有一个是我写页面时会碰到的。

每做一个这样的系统，我就多一块拼图。

不是为了名片上的头衔变成“产品经理”，是为了能用代码，去解决一个真实生意里、某个具体的人、某个具体的麻烦。

这一步，错题还只是被归了类。

等 AI 真的看懂照片、自己出错题那天，我再写一篇。

我是 daydayago，一个前端程序员，正在用 AI 把自己从写页面的人，重训成能做产品、能独立开发的人。
