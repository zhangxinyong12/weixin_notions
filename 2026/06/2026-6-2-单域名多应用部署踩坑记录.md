# 单域名部署多应用，我踩了一整天的坑

## 前言

昨天遇到个问题，把我折磨了一整天。

场景是这样的：**一个域名部署两个前端应用**

- 主应用（AA系统）：根路径 `/`
- 子应用（BB工具）：`/bb-tool` 路径

听起来很简单？实际上坑巨多。

今天把踩坑过程完整记录下来，包括 docker 配置、nginx 配置，还有那些我花了好长时间才搞明白的细节。

---

## 场景说明

### 部署架构

```
test.example.com (一个域名)
├── /         → 主应用（AA系统）
└── /bb-tool  → 子应用（BB工具）
```

### Docker 容器配置

docker-compose.yml：

```yaml
services:
  web-test:
    image: nginx:alpine
    container_name: multi-app-test
    volumes:
      # 主应用（根路径）
      - ./dist-main:/usr/share/nginx/html/main:ro
      # 子应用（/bb-tool 路径）
      - ./dist-sub:/usr/share/nginx/html/bb-tool:ro
      # nginx 配置
      - ./nginx-test.conf:/etc/nginx/nginx.conf:ro
    environment:
      - VIRTUAL_HOST=test.example.com
```

**目录结构**：
```
/usr/share/nginx/html/
├── main/          # 主应用构建产物
│   └── index.html
└── bb-tool/       # 子应用构建产物
    └── index.html
```

## 踩坑一：页面刷新跳回主应用

### 问题现象

- ✅ 从主应用点击链接，可以跳转到 `/bb-tool`
- ❌ 在 `/bb-tool` 页面刷新，会跳回主应用（根路径）

### 我的排查过程

#### 尝试 1：修改 nginx try_files

初始 nginx 配置：

```nginx
location / {
    root /usr/share/nginx/html/main;
    try_files $uri $uri/ /index.html;
}

location ^~ /bb-tool/ {
    root /usr/share/nginx/html;
    try_files $uri $uri/ /bb-tool/index.html;
}
```

我以为 try_files 写得不对，改成：

```nginx
location ^~ /bb-tool/ {
    try_files $uri $uri/ /bb-tool/index.html =404;
}
```

然后重启容器：

```bash
docker compose up -d web-test
```

**结果**：连跳转都不行了，点击链接直接跳回主应用。

#### 关键发现

我突然意识到：**我改 nginx 之前，至少能跳转过去；改完之后，连跳转都不行了**。

这说明我的修改方向错了。问题根本不在 nginx 的 try_files。

#### 真正的问题

问题在 **子应用的构建配置**。

子应用是用 Vite 构建的，vite.config.ts 里：

```typescript
const base = mode === 'test' ? '/bb-tool' : '/';
```

构建时的 base 路径是 `/bb-tool`，但这是个 SPA 应用。当用户直接访问 `/bb-tool` 或刷新时：

1. nginx 返回 `/bb-tool/index.html`
2. React Router **没有配置 basename**
3. 路由匹配失败，跳回根路径

**真正的原因**：vite改了base路径，但React Router忘记加basename了。

### 解决方案

在子应用的React Router中加basename：

```typescript
// App.tsx
import { BrowserRouter } from 'react-router-dom';

function App() {
  return (
    <BrowserRouter basename="/bb-tool">
      <Routes>
        <Route path="/" element={<Home />} />
        <Route path="/dashboard" element={<Dashboard />} />
      </Routes>
    </BrowserRouter>
  );
}
```

**关键**：
- vite的base路径：`/bb-tool`
- BrowserRouter的basename：`/bb-tool`
- 两者必须一致

这样配置后：
- 访问 `/bb-tool/dashboard` → nginx返回index.html → React Router匹配到 `/dashboard` → 渲染Dashboard组件
- 刷新 `/bb-tool/dashboard` → 同样能正常工作

### 最终方案

折腾半天后，我放弃了融合部署，改成了**分域名部署**：

- `test.example.com` → 主应用
- `test-main.example.com` → 主应用（备用域名）
- 单独给子应用分配域名

每个应用用自己的域名，互不干扰。

## 踩坑二：Docker 容器重启不生效

### 问题现象

修改 nginx 配置后，重启容器：

```bash
docker compose up -d web-test
```

但是配置不生效！

### 排查过程

#### 检查容器状态

```bash
docker compose ps
```

显示容器是 `Running`，但配置没更新。

#### 检查容器内的配置

```bash
docker exec multi-app-test cat /etc/nginx/nginx.conf
```

发现配置还是旧的。

### 真相

**docker compose up -d 不会重新挂载 volume**！

如果 volume 文件在宿主机改变了，容器内的文件不会自动更新。

### 解决方案

#### 方案 1：强制重建容器

```bash
docker compose up -d --force-recreate web-test
```

#### 方案 2：停止并删除容器

```bash
docker compose down web-test
docker compose up -d web-test
```

#### 方案 3：进入容器手动修改（不推荐）

```bash
docker exec -it multi-app-test sh
vi /etc/nginx/nginx.conf
nginx -s reload
```

**我用的方案**：方案 2，down 再 up。

## 踩坑三：一个 nginx 配置多个应用

### nginx 配置详解

```nginx
server {
    listen 80;
    server_name localhost;

    # ============================================
    # 主应用（根路径）
    # ============================================
    location / {
        root /usr/share/nginx/html/main;
        try_files $uri $uri/ /index.html;

        # 禁用缓存
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # ============================================
    # 子应用（/bb-tool 路径）
    # ============================================
    location = /bb-tool {
        return 301 /bb-tool/;
    }

    location ^~ /bb-tool/ {
        root /usr/share/nginx/html;
        try_files $uri $uri/ /bb-tool/index.html;

        # 禁用缓存
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # ============================================
    # CC检索 API 代理
    # ============================================
    location ^~ /api/cc {
        proxy_pass http://xxx.xxx.xxx.xxx:9004/api;

        # 超时配置
        proxy_connect_timeout 600s;
        proxy_send_timeout 600s;
        proxy_read_timeout 600s;
    }

    # ============================================
    # 其他 API 代理
    # ============================================
    location /api {
        proxy_pass http://xxx.xxx.xxx.xxx:9002;
    }
}
```

### 关键点

#### 1. root vs alias

```nginx
# 主应用用 root
location / {
    root /usr/share/nginx/html/main;
    # 访问 / → /usr/share/nginx/html/main/index.html
}

# 子应用用 root + 路径
location ^~ /bb-tool/ {
    root /usr/share/nginx/html;
    # 访问 /bb-tool → /usr/share/nginx/html/bb-tool/index.html
}
```

**区别**：
- `root`：会把 URI 路径附加到 root 路径后面
- `alias`：会替换掉 URI 路径

#### 2. location 优先级

```nginx
# 精确匹配
location = /bb-tool { ... }

# 前缀匹配（优先级高）
location ^~ /bb-tool/ { ... }

# 正则匹配
location ~ /bb-tool/ { ... }

# 普通前缀匹配
location /bb-tool/ { ... }
```

优先级：精确 > `^~` > 正则 > 普通。

#### 3. SPA 应用的 try_files

```nginx
try_files $uri $uri/ /index.html;
```

这个配置的作用：
1. 先找文件：`$uri`
2. 再找目录：`$uri/`
3. 都找不到，返回 `/index.html`（交给前端路由）

## 踩坑四：API 代理路径问题

### 场景

需要对接外部 CC 检索接口。

### 本地开发：CORS 跨域

直接调用外部接口，浏览器报 CORS 错误。

**解决方案**：vite proxy

```typescript
// vite.config.ts
server: {
  proxy: {
    '/api/cc': {
      target: 'http://xxx.xxx.xxx.xxx:9004',
      changeOrigin: true,
      rewrite: (path) => path.replace(/^\/api\/cc/, '/api'),
    },
  },
},
```

前端调用：

```typescript
fetch('/api/cc/search', { ... })
```

转换过程：
1. 浏览器请求：`http://localhost:5182/api/cc/search`
2. vite rewrite：去掉 `/api/cc`，变成 `/api/search`
3. 转发到：`http://xxx.xxx.xxx.xxx:9004/api/search`

### 线上部署：nginx 404

nginx 配置：

```nginx
location ^~ /api/cc {
    proxy_pass http://xxx.xxx.xxx.xxx:9004/api;
}
```

测试：

```bash
curl 'http://test.example.com/api/cc/search' \
  -H 'Content-Type: application/json' \
  --data-raw '{"query":"xxx"}'
```

返回 404。

### 排查过程

#### 尝试 1：修改 proxy_pass 路径

```nginx
# 写法 1
proxy_pass http://xxx.xxx.xxx.xxx:9004/api;

# 写法 2
proxy_pass http://xxx.xxx.xxx.xxx:9004/api/search;
```

都不行。

#### 尝试 2：对比 vite 配置

本地 vite 的逻辑：
- 前端请求：`/api/cc/search`
- rewrite 去掉 `/api/cc`，变成 `/api/search`
- 转发到：`http://xxx.xxx.xxx.xxx:9004/api/search`

nginx 的逻辑应该一样。

### 正确的 nginx 配置

```nginx
location ^~ /api/cc {
    proxy_pass http://xxx.xxx.xxx.xxx:9004/api;

    # 超时配置
    proxy_connect_timeout 600s;
    proxy_send_timeout 600s;
    proxy_read_timeout 600s;
}
```

转发逻辑：
- 前端请求：`/api/cc/search`
- nginx 去掉 location 匹配的部分 `/api/cc`
- 剩下 `/search`
- 拼接到 proxy_pass：`http://xxx.xxx.xxx.xxx:9004/api/search`

**关键点**：nginx 的 `proxy_pass` 末尾没有 `/` 时，会保留请求路径中 location 之后的部分。

## 总结

这次踩坑，几个关键点：

### 1. 单域名多应用部署

- Docker volume 映射要正确
- nginx location 优先级要搞清楚
- root 和 alias 的区别

### 2. Docker 容器重启

- `docker compose up -d` 不会重新挂载 volume
- 要用 `--force-recreate` 或先 down 再 up

### 3. SPA 应用路由

- try_files 要配置正确
- base 路径要和 nginx location 匹配
- 前端路由要处理刷新场景

### 4. nginx proxy 代理

- proxy_pass 末尾的 `/` 很关键
- location 匹配的部分会被去掉
- 剩余路径会拼接到 proxy_pass

### 5. 调试心态

- 改了之后更糟，立刻回退
- 本地正常线上炸，第一时间对比配置
- 环境差异要重视
