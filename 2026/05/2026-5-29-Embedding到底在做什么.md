# Embedding 到底在做什么

你调过 Embedding API 吗？

大概率调过。如果你做过 RAG、语义搜索、推荐系统，或者用过 LangChain、Chroma、Pinecone，你已经在用 Embedding 了。

但你有没有想过，调用一次 Embedding API，到底发生了什么？为什么一串数字就能表示"语义"？为什么余弦相似度能判断两段文字像不像？`king - man + woman = queen` 这个经典例子，真的靠谱吗？

这篇文章试着把 Embedding 这个东西讲透。不跳步骤，不假设你已经懂了。

---

## 第一步：调用 API，你拿到了什么

拿 OpenAI 的 `text-embedding-3-small` 举例。你传入一段文字，API 返回一个 JSON：

```json
{
  "object": "embedding",
  "embedding": [
    -0.006929283,
    -0.005336422,
    -0.000045471,
    -0.024047505
    // ... 还有 1532 个浮点数
  ]
}
```

1536 个浮点数。每个数大概在 -1 到 1 之间。

就这样。一段文字变成了一组数字。

这组数字有什么用？单独看，毫无意义。你不可能指着第 347 维说"这是情感维度"或者"这是主题维度"。每个维度都不是人类可解读的。

但 1536 个数放在一起，它们在做什么？它们在高维空间里标记了一个"位置"。

这个位置不是随便标的。两段意思相近的文字，它们在这个空间里的位置会很接近。两段意思不同的文字，位置会很远。

这就是 Embedding 的全部秘密：**把文字映射到高维空间中的一个点，使得语义相似 = 空间距离近。**

剩下的问题只有一个：怎么做到的？先不管怎么做到的，先确认"距离近"这件事怎么度量。

---

## 第二步：怎么算"距离近"

三个常用的度量方式：余弦相似度、点积、欧氏距离。

### 余弦相似度

两个向量之间的夹角的余弦值。

```
cos(a, b) = (a · b) / (||a|| × ||b||)
```

结果在 -1 到 1 之间。1 表示方向完全相同（最相似），0 表示正交（无关），-1 表示方向相反。

### 点积

直接把对应维度相乘再求和。

```
a · b = a₁b₁ + a₂b₂ + ... + aₙbₙ
```

### 欧氏距离

两个点之间的直线距离。

```
d(a, b) = √((a₁-b₁)² + (a₂-b₂)² + ... + (aₙ-bₙ)²)
```

### 关键关系

OpenAI 的 Embedding 输出是**归一化**的——每个向量的长度（模）等于 1。这意味着：

- 余弦相似度 = 点积（数学上完全等价）
- 欧氏距离² = 2 - 2 × 点积

三种度量算出来的排序**一模一样**。所以用哪个都行，点积最快。

### 一个具体的 3D 例子

假设 a = [0.5, 0.8, 0.3]，b = [0.6, 0.7, 0.4]：

```
点积 = 0.5×0.6 + 0.8×0.7 + 0.3×0.4 = 0.30 + 0.56 + 0.12 = 0.98

||a|| = √(0.25 + 0.64 + 0.09) = 0.990
||b|| = √(0.36 + 0.49 + 0.16) = 1.005

余弦相似度 = 0.98 / (0.990 × 1.005) = 0.985
```

0.985，非常接近 1。这两个向量方向几乎一致，语义上可以认为很相似。

当然，这是 3 维的情况。真实的 Embedding 是 1536 维。但计算方法完全一样——只是从 3 个维度变成 1536 个维度的累加。

## 第三步：king - man + woman = queen，是真的吗

你可能看过这个经典的例子：

> king 的向量 - man 的向量 + woman 的向量 ≈ queen 的向量

这个故事在无数文章和教程里出现过。但它实际上有多靠谱？

Mike X Cohen 做过一个真实测试，用 BERT 的 Embedding 跑了这个计算。结果：

| 候选词 | 与计算结果的相似度 |
|--------|-----------------|
| queen | 0.568 |
| kings | 0.400 |
| princess | 0.390 |

queen 确实排第一。但 0.568 的相似度，意味着只有 28.4% 的共享方差。这是一个微弱的信号，不是一个确凿的结论。

换一个类比试试。"tree 之于 leaf，好比什么之于 petals？"预期答案是 flower。实际结果：

| 候选词 | 相似度 |
|--------|--------|
| trees | 0.403 |
| 名（汉字）| 0.295 |
| balloons | 0.289 |

完全不对。

**结论**：king/queen 这个例子是精心挑选的特例。它不代表 Embedding 真的理解了"性别"这个概念，更不代表你可以用向量加减法做推理。Embedding 能做到的是"语义相近的东西离得近"，仅此而已。

---

## 第四步：为什么语义相近的东西会离得近

这才是 Embedding 的核心问题。

答案是训练过程。

Embedding 模型用一种叫**对比学习（Contrastive Learning）** 的方法训练。思路很简单：

1. 准备大量"相似对"和"不相似对"的文本
2. 模型的目标：让相似对的向量靠近，不相似对的向量远离
3. 反复训练，直到向量空间被组织成"语义近 = 距离近"

具体怎么做？一种常见的方法是 InfoNCE Loss：

```
L = -log(exp(sim(anchor, positive) / τ) / Σ exp(sim(anchor, negative_i) / τ))
```

用人话说：给定一段文本（anchor），和它语义相似的一段文本（positive），以及一堆不相关的文本（negatives），要求模型把 anchor 和 positive 的相似度拉高，同时把 anchor 和所有 negatives 的相似度压低。

每个训练步骤都在回答一个问题："这两段文字是真正相关，还是只是碰巧有几个词一样？"

训练数据从哪来？来源很多：问答对（StackOverflow 的 Q&A）、引用关系（论文 A 引用了论文 B）、同义句（翻译数据）、人工标注的相似对。模型在几亿对文本上训练完之后，向量空间就被组织好了。

所以 1536 个维度里装的不是某个"概念"，而是一种**压缩表示**。模型把所有它见过的语义信息，压缩到了 1536 个浮点数里。每个维度可能同时编码了多个语义特征的组合，而且这些组合是人类无法解读的。

## 第五步：Embedding 做不到的事

了解了原理之后，就容易理解 Embedding 的局限了。

### 它不理解否定

> "治疗改善了患者的预后" vs "治疗**没有**改善患者的预后"

这两句话的 Embedding 相似度：**0.96**。

几乎一模一样。因为两句话的大部分文字相同，Embedding 模型没有真正理解那个 "not" 在翻转整句话的意思。

> "好" vs "不好" → 相似度 0.88
> "极好" vs "极其不好" → 相似度 0.92

这是一个在生产环境中会咬你的坑。如果用户搜"不带触摸屏的笔记本"，向量检索很可能返回带触摸屏的笔记本——因为"不带触摸屏"和"带触摸屏"的向量太像了。

解决方案：混合搜索。用 BM25 关键词检索补充精确匹配。

### 它不懂数字

> "投资年回报率 2%" vs "投资年回报率 20%"

相似度：**0.97**。对 Embedding 来说，"2%"和"20%"几乎一样。

如果你要做精确的数值比较（价格区间、日期范围、数量筛选），不要指望 Embedding。用结构化查询（SQL、过滤条件）。

### 不同模型的 Embedding 不能混着比

同一段文字，用 `text-embedding-3-large` 和 `text-embedding-3-small` 分别 Embedding，算余弦相似度：**-0.001**。

完全不相关。因为不同模型的向量空间不一样，就像两套不同的坐标系，你不能拿一个坐标系里的坐标去跟另一个坐标系里的坐标比距离。

换模型 = 重新 Embedding 整个语料库。没有捷径。

## 第六步：1536 维是不是太多了

OpenAI 的 `text-embedding-3-small` 输出 1536 维，`text-embedding-3-large` 输出 3072 维。每条数据占 1536 × 4 字节 = **6 KB**（float32）。100 万条就是 6 GB。

但 OpenAI 支持**缩短维度**。你可以在 API 调用时指定 `dimensions` 参数：

```python
response = client.embeddings.create(
    model="text-embedding-3-small",
    input="你好世界",
    dimensions=256  # 只取前 256 维
)
```

为什么截断不会严重损失质量？因为 OpenAI 用了一种叫 **Matryoshka Representation Learning（MRL）** 的训练方法。它的做法是：在训练时，同时对多个截断长度（64、128、256、512、768...）计算 Loss，然后加在一起做反向传播。这迫使模型把最重要的信息集中在靠前的维度里。

实测数据（OpenAI 官方基准）：

| 配置 | 检索质量 |
|------|---------|
| text-embedding-3-large @ 3072 维 | 满分 |
| text-embedding-3-large @ 256 维 | **超过老模型 ada-002 @ 1536 维** |
| text-embedding-3-large @ 1024 维 | 接近满分 |

从 3072 维砍到 256 维，存储减少 **12 倍**，质量只掉了不到 1%。而 256 维的 large 模型比 1536 维的老模型还要好。

这就是为什么 OpenAI 说维度是可调的——因为训练的时候就设计好了。

**注意**：如果你手动截断（不是通过 API 参数），截断之后必须重新归一化：

```python
def truncate_and_normalize(embedding, target_dims):
    truncated = embedding[:target_dims]
    norm = np.linalg.norm(truncated)
    return truncated / norm
```

不归一化的话，余弦相似度的计算结果会偏。

## 第七步：模型怎么选

2025-2026 年主流 Embedding 模型对比：

| 模型 | 维度 | 价格/百万token | 精度 | 特点 |
|------|------|--------------|------|------|
| OpenAI text-embedding-3-small | 1536 | $0.02 | 62.3% | 安全默认选择 |
| OpenAI text-embedding-3-large | 3072 | $0.13 | 64.6% | 贵 6 倍，高 2 分 |
| Jina-embeddings-v3 | 1024 | $0.02 | 65.5% | 同价精度最高 |
| Voyage-3-large | 1024 | $0.18 | 67.1% | 精度最高，贵 |
| Gemini Embedding 2 | 768 | 免费 | 多项第一 | 免费且全能 |
| GTE-large（开源） | 1024 | 自部署免费 | 65.4% | 开源最优 |
| BGE-M3（开源） | 1024 | 自部署免费 | 63.6% | 多语言强 |

几个判断：

**价格敏感选 Jina-v3。** 跟 OpenAI small 同价，精度高 3 个点。性价比最高。

**要免费选 Gemini Embedding 2。** Google 的模型，在跨语言、关键信息检索两项上都是第一。而且免费。

**要最高精度选 Voyage-3。** 但价格是 Jina 的 9 倍，除非你有极致的检索质量要求，否则不值。

**要自部署选 GTE-large。** 阿里达摩院出品，开源模型里精度最高，支持 8192 token 长文本。

但说实话，**模型之间的差距远没有"怎么用"重要**。一个便宜的模型配上好的分块策略和混合搜索，效果远好于最贵的模型配上默认的分块和纯向量检索。

## 总结

回到最初的问题：Embedding 到底在做什么？

它在做一件事：**把语义信息压缩成一组数字，使得语义相似 = 数字距离近。**

这组数字是怎么来的？通过对比学习训练出来的。模型在几亿对"相似/不相似"的文本上反复训练，学会了怎么组织这个向量空间。

它不是魔法。它不能推理，不理解否定，不懂数字，不同模型之间不能混用。

但在这个能力范围内，它是一个极其有效的工具。几乎所有语义搜索、RAG、推荐系统的底层都在用它。

理解了它在做什么、怎么做的、做不了什么，你在用 LangChain、Chroma、Pinecone 这些工具的时候，就不只是在调 API 了。
