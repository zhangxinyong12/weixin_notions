# 做了几个 RAG 项目后，我发现大家都在优化错的地方

我给客户搭过四个知识库。

第一个是个电商客服 FAQ，500 条问答，半天搞定，向量检索就够用，客户很满意。

第二个开始翻车。

---

## 第一个教训：答错了不是模型的锅

第二个项目是给一家小公司搭内部知识库。需求听着简单：把公司文档扔进去，员工用自然语言提问，AI 回答。

我搭好了，跑了一组测试题。大部分问题答得不错。然后老板随手问了一句："老王负责的那个项目的预算是多少？"

AI 回了一堆数字。看着挺自信。但全是错的。

我去查，发现问题出在文档被切成碎片之后，"老王"在一段里，"项目A"在另一段里，"预算50万"又在第三段里。向量检索分别找到了这些片段，但它不知道老王和项目A之间有个"负责"关系，也不知道项目A和50万之间有"预算"关系。

就像你问一个人"你同事的宠物叫什么"，但他只有一沓乱序的便签纸，每张纸写了一句不相关的话。能答对才怪。

我之前写过用 GraphRAG 解决这个问题的过程。但这次想聊另一个事。

---

## 73% 的失败出在检索上

后来我在网上查 RAG 的生产数据，看到一个数字：**RAG 系统答错的时候，73% 的原因是检索环节没找对内容**，不是模型生成得不好。

这个数字跟我自己的经历对上了。四个项目里，我没遇到一次"模型不够聪明"的问题。每次翻车，都是因为喂给模型的东西不对。

换句话说，你在 GPT-4o 和 Claude 之间纠结选哪个，不如把时间花在分块策略上。

我后来做了个对比测试：

| 分块方式 | Context Recall（需要的信息被检索到的比例） |
|---------|------------------------------------------|
| 固定 512 token 切 | 0.72 |
| 句子窗口（检索到句子后扩展到前后3句） | 0.88 |

从 0.72 到 0.88，意味着每 10 个问题，多找回差不多 2 个之前漏掉的。这个提升比你换一个更贵的模型大得多。

---

## 分块：RAG 里最被低估的环节

大部分 RAG 教程教你选向量数据库、选 embedding 模型，分块就用默认的 512 token 一切完事。我一开始也是这么干的。后来发现分块才是整个链路里最要命的环节。切错了，那段信息从根上就丢了，后面再怎么优化都补不回来。

我后来踩过的几个分块的坑：

**1. 固定分块会切断语义边界。**

一段话说"合同规定，乙方应在收到甲方通知后30日内完成交付。但如果是不可抗力导致延迟，乙方可以申请延期，最长不超过60日。"如果正好在"但"字前面被切断，上半段说的是"30日交付"，下半段说的是"延期60日"。AI 如果只拿到上半段，就会给出一个错误的、缺少关键例外的答案。

更可怕的是，这个错误看起来很合理。没有报错，没有异常。用户以为是对的。

**2. 表格是隐形杀手。**

法律文档那个项目里，客户有一堆产品价格表、供应商信息表。这些表格被切分之后变成了什么？"Product A, Region EMEA, Q3 Revenue 4.2M, YoY Growth 12%"——没有表头，AI 根本不知道这些数字是什么意思。

我后来的做法：在索引阶段就把表格的每一行转成自然语言。"产品A在欧洲中东非洲地区Q3营收420万，同比增长12%。"土，但管用。

**3. 一种分块策略不能走天下。**

后来我的做法是按文档类型走不同的分块策略：

```python
def get_parser(doc):
    doc_type = doc.metadata.get('doc_type', 'unknown')
    if doc_type in ['spec', 'contract', 'runbook']:
        # 结构化文档：层级分块（2048 → 512 → 128）
        return HierarchicalNodeParser.from_defaults(
            chunk_sizes=[2048, 512, 128]
        )
    elif doc_type in ['policy', 'faq', 'guide']:
        # 叙述性文档：句子窗口
        return SentenceWindowNodeParser.from_defaults(window_size=3)
    else:
        # 兜底：固定分块
        return TokenTextSplitter(chunk_size=512, chunk_overlap=50)
```

不优雅。但四个项目下来，我觉得在分块上"不优雅但针对性"比"优雅但一刀切"强得多。

---

## 混合搜索：投入产出比最高的优化

如果说分块是"怎么切"，那检索就是"怎么找"。

我最早的方案是纯向量检索：把问题转成向量，在向量数据库里找最近的几段。大部分场景够用。但有些问题就是找不到。

比如用户问"合同编号 CT-2024-087 的违约金条款"。纯向量检索可能找到一堆"违约金"相关的段落，但不是编号 CT-2024-087 那份合同的。因为向量检索是按"语义相似度"找的，它理解"违约金"是什么意思，但它不擅长精确匹配"CT-2024-087"这种具体编号。

后来我加了 BM25 关键词检索，跟向量检索并行跑，然后用 Reciprocal Rank Fusion（RRF）合并结果。效果立竿见影。

BM25 擅长精确匹配（合同编号、人名、产品名），向量检索擅长语义匹配（"老王负责的项目预算"理解成"找老王+项目+预算"）。两个一起用，互相补短板。

我的判断：如果你只有一个优化预算，加混合搜索。别换模型，别换 embedding，先加 BM25。这是投入产出比最高的一个改动。

---

## Reranker：锦上添花还是雪中送炭

检索到 10 段内容之后，我加了一步 Reranker（用的是 BGE-Reranker）。它的作用是对检索结果重新打分排序，把最相关的排到前面。

实测数据：不加 Reranker，回答准确率大概 70%。加了之后，85% 左右。

Anthropic 做过一个更系统的测试。他们用了一种叫 Contextual Retrieval 的方法——在每个分块前面加一小段上下文说明（用小模型生成），让每个分块不那么孤立。检索失败率降低了 49%。再加上 Reranker，降低 67%。

Reranker 不是必须的。FAQ 类项目，检索结果本来就不多，加不加差别不大。但文档多、查询复杂的场景（法律、合规、内部知识库），这一步很值。

---

## 你的 RAG 可能在"骗你"

有个测试方法我觉得特别好：把检索到的内容去掉，让模型直接回答同一个问题。如果答案差不多，说明你的检索可能白做了。加了一堆延迟和基础设施，但模型根本没用上你检索的东西，它在用自己的训练数据回答。

这个测试帮我发现过一个问题：电商 FAQ 那个项目，我后来回头测了一下，发现去掉检索之后，GPT-4o-mini 对常见电商问题的回答质量几乎没变。500 条 FAQ，大部分本来就是通用知识。真正依赖检索的可能就 50 条。

如果重来，我会先做这个测试，再决定要不要上 RAG。

---

## 什么时候不该用 RAG

这个判断很少有人写，但我觉得比任何技术选型都重要。

**知识库小于 500 页的，别上 RAG。** 直接把全部内容塞进 prompt，用 Anthropic 的 prompt caching 或者类似机制。200,000 token 以内的知识库，塞进 prompt 比 RAG 更准、更快、更便宜。RAG 是为"塞不下"的场景准备的。

**数据在数据库里的，用 SQL。** "Q3 营收多少"这种问题，直接 `SELECT revenue FROM financials WHERE quarter='Q3'` 就完事了。让 AI 生成 SQL 去查，比把财务报表切成文本块再让向量检索去猜，靠谱一万倍。

**内容变化特别快的，想清楚再上。** RAG 的 embedding 不是实时更新的。如果你的文档每天改几十次（比如库存、价格），要么做流式索引（文档一入库就立即可查），要么干脆别用 RAG，直接查数据库。

我电商 FAQ 那个项目，500 条 Q&A，如果现在重新做，我大概率不用 RAG。直接 prompt caching 塞进去，又准又省事。但当时不懂，走了弯路。

## Embedding 模型怎么选

最后说一个大家纠结的问题。我做过的测试：

| 模型 | 价格（每百万token） | 精度（MTEB） | 我的评价 |
|------|-------------------|-------------|---------|
| OpenAI text-embedding-3-small | $0.02 | 62.3 | 安全默认选择 |
| Jina-embeddings-v3 | $0.02 | 65.5 | 同价格精度更高，推荐 |
| Voyage-3-large | $0.18 | 67.1 | 精度最高，贵 |
| OpenAI text-embedding-3-large | $0.13 | 64.6 | 贵了6倍，只高2分 |
| GTE-large（开源） | 免费 | 65.4 | 开源里最好的 |
| BGE-large（开源） | 免费 | 63.6 | 稳定 |

我的选择：**Jina-v3。** 跟 OpenAI small 一样便宜，但精度高 3 个点。如果需要本地部署，用 GTE。Voyage 精度最高，但价格是 Jina 的 9 倍，我觉得不值。

但说实话，embedding 模型之间的差距，远没有分块策略的差距大。换模型能提升 3-8% 的召回率，换个分块策略能提升 15-20%。精力该花在哪，自己算。

## 我现在搭 RAG 的顺序

四个项目下来，我形成了一个固定的顺序：

1. 先问：这个数据量真的需要 RAG 吗？小于 500 页的直接 prompt caching。
2. 再问：问题是"查数据"还是"查文本"？查数据走 SQL，查文本才走 RAG。
3. 按文档类型选择分块策略，别一刀切。
4. 先跑纯向量检索，测一遍。
5. 加 BM25 混合检索，再测一遍。这是我见过提升最大的一步。
6. 文档多、查询复杂的话，加 Reranker。
7. 做"去检索对照测试"：去掉检索看答案变不变。不变说明检索在装样子。

模型选型放在最后。先把分块和检索搞好，再考虑要不要换更贵的模型。

这也是我跟客户沟通时常碰到的一个误解。很多人觉得 RAG 的核心是模型要够聪明。但说实话，模型聪明不聪明没那么重要。重要的是你把什么喂给了它。
