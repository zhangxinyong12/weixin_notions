# 做了 5 个知识库项目后，我整理了一份 RAG 选型终极指南

> 向量 RAG、GraphRAG、LightRAG、Vectorless RAG……到底选哪个？一个前端程序员的实战总结。

---

上周有个朋友问我："我想给公司搭个知识库，该选什么方案？"

我说你等等，这个问题我花了三个月才搞明白，让我从头讲。

---

## 先搞清楚：RAG 是什么

大模型很聪明，但它不知道你公司的产品文档、不知道你的业务数据。**RAG（检索增强生成）** 就是让大模型在回答之前，先去你的知识库里查资料，然后基于查到的内容回答。

核心流程三步：

```
用户提问 → 从知识库检索相关内容 → 大模型基于检索结果生成回答
```

问题在于——"知识库"这三个字，背后至少有五种完全不同的技术路线。

---

## 第一种：向量 RAG（最经典）

这是 90% 的人入门 RAG 的方式，也是目前最成熟的方案。

**原理**：把文档切成小块，用 Embedding 模型转成向量（一组数字），存到向量数据库里。用户提问时，把问题也转成向量，找最"相似"的文档块，喂给大模型。

```
文档 → 切块 → Embedding → 向量数据库
                              ↑
用户提问 → Embedding → 相似度匹配 → 取出 Top-K → LLM 生成回答
```

**优点**：

- 上手快，LangChain 几十行代码就能跑
- 适合大规模非结构化文档
- 生态成熟，工具链完善

**致命短板**：

- **多跳推理不行**。问"老王负责的项目预算是多少"，如果"老王"、"项目"、"预算"分布在不同段落，向量检索可能拼不出完整答案
- **上下文丢失**。文档切块后，块与块之间的关系没了
- **语义相似≠答案相关**。有时候检索到的文档"看起来像"，但不是真正需要的

**适合场景**：FAQ 问答、客服机器人、文档搜索、内容总结

---

## 第二种：GraphRAG（2025-2026 最热）

既然向量 RAG 的死穴是"关系"，那就把关系存下来——用**知识图谱**。

**原理**：先让 LLM 从文档中抽取实体和关系，构建知识图谱。查询时，沿着图谱的关系链路找到答案。

```
老王 --[负责]--> 项目A --[预算]--> 50万
  |                                  |
  +--[所属]--> 技术部                 |
                                      
问："老王负责的项目预算？" → 沿图谱遍历 → 50万
```

微软的 GraphRAG 论文（2024 年发布）是这波浪潮的起点。它的核心创新是**社区检测**——把图谱分层，每个社区生成一段摘要，这样既能回答细节问题，也能回答全局问题。

**优点**：

- 多跳推理能力碾压向量 RAG
- 能回答全局性问题（"所有项目的预算分布是什么？"）
- 回答可解释，能追溯推理路径
- 准确率高：微软报告称 GraphRAG 在全面性上比传统 RAG 高 72%-83%

**致命短板**：

- **成本高**。构建知识图谱需要大量 LLM 调用来抽取实体和关系
- **更新难**。微软 GraphRAG 的社区结构是全局计算的，新增文档后需要**重建整张图**
- **维护门槛高**。需要懂知识图谱、Cypher 查询语言
- **查询速度慢**。图遍历比向量检索慢不少

**适合场景**：法律研究、医疗诊断、复杂关系查询、学术研究、多文档关联分析

---

## 第三种：LightRAG（GraphRAG 的轻量替代）

GraphRAG 太重了？LightRAG 就是来"减肥"的。

**原理**：同样基于知识图谱，但做了三个关键简化：

1. **不划分社区**，采用**双层检索**（实体层 + 关系层）代替社区分层
2. **支持增量更新**，新文档入库时只更新局部图谱，不用重建整张图
3. **去重机制**，相同实体自动合并，图谱不会越建越臃肿

**和 GraphRAG 的核心区别**：

| 维度 | GraphRAG | LightRAG |
|------|----------|----------|
| 检索方式 | 社区分层检索 | 双层检索（实体+关系） |
| 更新方式 | 重建整张图 | 增量更新 |
| 构建成本 | 高（需要全局社区计算） | 低（直接构建） |
| 查询速度 | 较慢（全局聚合） | 快（直接检索） |
| 全局问答 | 强（社区摘要） | 一般（无社区结构） |

**优点**：

- 比 GraphRAG 便宜很多，构建速度快
- 支持增量更新，适合数据频繁变化的场景
- 查询效率高

**致命短板**：

- 全局性问答能力不如 GraphRAG（没有社区摘要）
- 图谱质量依赖抽取模型的水平
- 比较新，生产环境验证不够充分

**适合场景**：企业知识库（文档经常更新）、动态数据问答、中等复杂度关系查询

---

## 第四种：PathRAG（路径感知的图检索）

这是图 RAG 家族里的新成员，思路很独特——不关注社区，关注**路径**。

**原理**：在知识图谱上，找到从查询实体到答案实体之间的"路径"，用路径上的信息作为上下文。

```
问："治疗X疾病的药物有哪些副作用？"

X疾病 --[治疗药物]--> 药物A --[副作用]--> 头痛
                  |--> 药物B --[副作用]--> 恶心
                  
→ 沿路径检索：[X疾病→药物A→头痛] + [X疾病→药物B→恶心]
```

**优点**：

- 路径推理能力强，适合"从 A 到 B 经过什么"这类问题
- 增量友好，新数据不需要全局重建
- 中等成本

**适合场景**：供应链追溯、因果推理、路径类查询

---

## 第五种：Vectorless RAG（2026 新范式，连向量都不要了）

这是 2026 年最激进的方向——**完全抛弃向量数据库**。

**原理**：不切块、不向量化、不存向量数据库。让 LLM 直接"阅读"文档结构，基于推理来定位需要的信息。代表框架是 **PageIndex**。

它的核心观点是：**分块本身就是有损操作**。文档被切块后，层次结构、交叉引用、上下文关系、视觉排版全丢了。向量检索再怎么优化，都是在"残缺数据"上做匹配。

**流程**：

```
文档 → 保留原始结构（标题、段落、表格、引用关系）
                            ↑
用户提问 → LLM 推理定位 → 直接在文档结构中找到相关部分 → 生成回答
```

**优点**：

- 不需要向量数据库，架构简单
- 保留了文档的完整上下文和层次结构
- 适合复杂长文档（合同、论文、技术规范）

**致命短板**：

- 非常新，2026 年初才兴起，生态不成熟
- LLM 调用成本高（每次查询都要推理）
- 不适合海量文档（没有向量数据库的索引加速）
- 实际生产案例很少

**适合场景**：复杂长文档问答、合同审查、学术论文分析

## 选型决策树

说了这么多，到底怎么选？我用一个决策树总结：

```
你的知识库是什么类型？
│
├─ 大量非结构化文档（FAQ、产品手册、新闻）
│  └─ 选向量 RAG，够用且成熟
│
├─ 文档之间有复杂关系（法律、医疗、供应链）
│  ├─ 需要全局性问答（"所有案件的趋势"）
│  │  └─ 选 GraphRAG
│  │
│  ├─ 数据经常更新，追求性价比
│  │  └─ 选 LightRAG
│  │
│  └─ 主要查询实体间的路径/因果链
│     └─ 选 PathRAG
│
├─ 少量但非常复杂的文档（合同、论文、技术规范）
│  └─ 试试 Vectorless RAG（但做好踩坑准备）
│
└─ 都有？复杂关系 + 大量文档 + 需要兼顾
   └─ 混合架构：向量检索 + 图谱增强（生产推荐）
```

## 生产环境推荐：混合架构

如果你要做一个真正的生产系统，我建议**不要只选一种**。

我在接单项目中的实际做法是：

```
用户提问
  │
  ├─ 第一步：向量检索，快速找到相关文档块（广撒网）
  │
  ├─ 第二步：知识图谱查询，补充关系信息（精准补全）
  │
  ├─ 第三步：Reranker 重排序（推荐 BGE-Reranker）
  │
  └─ 第四步：LLM 综合生成回答
```

**为什么要加 Reranker？**

向量检索找的是"相似的"，但相似不等于"最需要的"。Reranker 是一个专门的排序模型，会对初步检索结果做二次打分，把最相关的排到前面。

我用 BGE-Reranker 之后，回答准确率从 70% 提升到了 85% 以上。这是投入产出比最高的一步优化。

## 向量数据库怎么选

如果你用了向量 RAG 或混合架构，向量数据库的选择也很关键。2025-2026 年主流方案：

| 数据库 | 一句话定位 | 适合谁 |
|--------|-----------|--------|
| **Milvus** | 企业级分布式，支持千亿级向量 | 大规模生产系统，QPS 百万级 |
| **Qdrant** | Rust 写的，高性能低延迟 | 追求性能的中等规模场景 |
| **Chroma** | Python 原生，极简上手 | 个人项目、原型验证 |
| **Pinecone** | 全托管 Serverless | 不想运维、预算充足 |
| **Weaviate** | 语义+结构搜索，内置 ML | 需要混合检索的场景 |
| **PGVector** | PostgreSQL 插件 | 已有 PG 基础设施的团队 |

**我的建议**：

- 学习/接单小项目：**Chroma**，10 行代码就能跑
- 中型生产系统：**Qdrant**，性能好、部署简单
- 大型企业系统：**Milvus**，分布式能力强
- 不想管服务器：**Pinecone**，花钱买省心

## 分块策略也很关键

选好了方案，还有一个容易忽略的环节：**文档怎么切块**。

2026 年主流的分块策略：

| 策略 | 原理 | 适用场景 |
|------|------|---------|
| **固定大小** | 按字符数切 | 简单场景，快速验证 |
| **递归分割** | 按段落→句子→字符逐级切 | LangChain 默认方案，通用 |
| **语义分块** | 按语义边界切 | 内容主题变化多的文档 |
| **Late Chunking** | 先整篇向量化，再切块 | 保留全局上下文 |
| **Contextual Retrieval** | 每个块前加文档级摘要 | Anthropic 推荐，效果好 |

**Late Chunking** 是 2025-2026 年的新突破——传统方法是先切块再向量化，每个块丢失了全局上下文。Late Chunking 反过来：先把整篇文档向量化，再在向量空间中切分。这样每个块的向量都"记得"整篇文档的上下文。

实测效果：在复杂文档上，Late Chunking 比传统分块的检索准确率提高了 15%-30%。

**我的实战建议**：先用递归分割跑起来，遇到效果不好的场景再换 Late Chunking 或 Contextual Retrieval。别一上来就追求最优解。

## 图数据库选 Neo4j 还是别的？

如果选了 GraphRAG / LightRAG / PathRAG，图数据库的选择：

| 数据库 | 特点 |
|--------|------|
| **Neo4j** | 最主流，LangChain 原生支持，社区资源最多 |
| **NebulaGraph** | 国产，分布式架构，适合大规模 |
| **TigerGraph** | 企业级，分布式查询性能强 |

**我选 Neo4j**——不是因为最好，而是因为生态最完善。LangChain 的 GraphRAG 教程全是基于 Neo4j 的，踩坑的人多意味着你遇到问题能找到答案。

## 我自己的选型经历

说说我实际做的几个项目，给你参考：

**项目 1：电商客服 FAQ**

- 数据：500 条产品 Q&A
- 方案：向量 RAG + Chroma
- 效果：够用，半天搞定
- 成本：几乎为零

**项目 2：公司内部知识库（就是我上篇文章写的）**

- 效果：增量更新很方便，每周新增 20-30 份文档
- 成本：中等，构建图谱的 LLM 调用费用约 200 元

## 最后：别过度设计

说句大实话——大部分知识库项目，**向量 RAG 就够了**。

很多人一上来就想搞 GraphRAG、知识图谱，结果花了三周还没上线。我的建议是：

1. **先用向量 RAG 跑起来**，验证需求
2. 遇到向量 RAG 解决不了的问题（多跳推理、全局问答）
3. **再叠加图谱层**，形成混合架构
4. 每加一层，都要有明确的收益衡量

技术选型不是选最好的，是选**最适合当前阶段**的。

能 80 分的方案两天上线，好过 95 分的方案两个月还没影子。先让系统跑起来，再迭代优化——这才是接单和做产品的正确姿势。
