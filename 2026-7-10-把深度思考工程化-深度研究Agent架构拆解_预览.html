AI Agent · 架构深度拆解

做了一年 AI Agent，我终于把“深度思考”工程化了

复杂问题答不好、汇总像拼接、过程全黑盒——根因都是同一个：缺一个“规划层”。这篇拆解一个真正能处理复杂问题的深度研究 Agent，是怎么一步步搭起来的。不讲概念，讲架构。

一、我的 Agent，一遇复杂问题就拉胯

简单问题，单 Agent 秒杀：查个数据、翻译一段、提取个指标。但一旦用户问——

“分析这家公司未来三年的核心风险，并判断它值不值得投资。”

系统就开始崩，而且是四种崩法：

① 漏：只识别出一个主意图，子问题全丢。

② 乱：一堆 Agent 同时被触发，谁先谁后没人管。

③ 散：各答各的，没有统一标准，也没有互相校验。

④ 糊：最后的汇总，就是把几段输出粘贴在一起。

我折腾很久才看清根因：我让路由器（Router）去理解整个复杂问题，它扛不动。Router 擅长的是“为一个明确的任务选执行者”，不是“拆解一个发散的大问题”。

二、第一个认知转弯：先判断“问题粒度”，再谈意图

老链路是问题 → 意图识别 → 单个 Agent。它适合边界清楚的简单问题。复杂问题的正确姿势是：先分诊，再决定要不要规划。

简单问题 → 直接 Router → 单 Agent → 答案
复杂问题 → 先拆成任务 → 再 Router 逐个路由 → 多 Agent 执行 → 汇总校验

注意：意图识别没有被淘汰，它是被降级了——从“理解整个用户问题”，变成“为已经拆好的单个任务，选一个执行者”。这是它能做好的事。

一句话原则
保留你的意图路由，但别再让它直接扛复杂问题；先让 Planner 把问题拆成任务，再让 Router 对任务做路由。

三、一个深度研究 Agent，其实有九个节点

把上面的原则展开，完整链路长这样。每个节点只干一件事：

Query Triage 分诊 — 这问题简单还是复杂？要不要规划？

Problem Framer 定义问题 — 用户到底要做什么决策？边界在哪？

Planner 拆任务 — 拆成 3-8 个子任务，声明依赖。

Task DAG 任务图 — 带依赖的任务列表，决定谁能并行。

Task Router 任务路由 — 为单个任务选 Agent（复用你的老路由）。

Agent 执行 — 专业 Agent 干活，返回统一格式。

Reflector 反思 — 覆盖全不全？有冲突或缺证据吗？要补任务吗？

Synthesizer 汇总 — 围绕原问题综合，不是粘贴。

Evaluator 验收 — 答案完整、可信吗？过不了打回。

看着多，其实核心就一句话：拆开、分工、查漏、收口。这四步任何一个做不好，复杂问题就答不好。

四、五个让系统“立得住”的关键设计

01Planner 只写“能力需求”，不写死 Agent 名

Planner 输出的是 required_capabilities（需要什么能力），绝不是 environment_agent 这种具体名字。为什么？因为规划层和执行层必须解耦——Agent 换了、升级了，计划不用改。耦合了，整个系统就脆。

02所有 Agent 走统一证据协议

每个 Agent 必须返回：conclusion（结论）/ findings（发现）/ evidence（证据）/ assumptions（假设）/ gaps（信息缺口）/ confidence（置信度）。而且每条结论要区分事实 fact / 推断 inference / 假设 assumption，并挂上 evidence_ids。没这个，汇总就是一锅粥，更别提审计。

03Reflector 必须严格受限

“做完反思一下、不够再补”听起来很美，但不受限的反思会让成本无限膨胀。所以必须卡死：初始任务 3-8 个、最大深度 3 层、最多跑 2-3 轮、单轮新增任务不超过 3 个。只在“高度相关 + 可能改变结论 + 现有任务没覆盖 + 预算内”四条同时满足时才加任务。

04Synthesizer 是综合器，不是拼接器

两条铁律，是区分“研究”和“糊弄”的底线：一是不因为多个 Agent 重复同一个观点，就提高置信度（三个 Agent 都说它好，不等于证据更强）；二是来源冲突不能靠多数投票消解，消解不了必须原样披露。汇总要围绕原始问题重新组织，不是按 Agent 顺序粘贴。

05不给用户看思维链，只记录结构化规划

模型的逐字推理过程，既不可控也不可复现，不该写进日志或展示给用户。系统真正该记录的，是结构化的规划信息：目标、子问题、执行计划、任务依赖、证据缺口。给用户看的，只有高层进度——“已生成 5 个研究任务”“正在并行分析”“正在核查证据”。

五、背后是一组经典模式，别被名词吓住

这套架构不是凭空发明的，它站在几个成熟模式上：

Plan-and-Execute：先规划，后执行。

ReWOO：计划里显式声明任务依赖和中间结果引用，不用每步都重新想。

LLMCompiler：把计划编译成 DAG，无依赖的任务并行跑。

Supervisor / Worker：总控调度多个专业 Agent。

Reflector / Evaluator-Optimizer：执行后查缺口，必要时局部补。

一个常见误解：以为这些是要装的三件套。其实——LangGraph 是框架，ReWOO／LLMCompiler／Reflector 是设计模式。你只需要用 LangGraph 把这些思想实现出来就够了，根本不用把它们都引进来。

想抄作业，可以看这几个开源：Open Deep Research（最接近，首选）、DeerFlow（字节的，Planner/Reporter 组织得清楚）、GPT Researcher（拆解+来源+报告）、STORM（多视角问题树）、Magentic-One（Orchestrator 进度管理）。

六、它和“大模型深度思考”的本质区别

用户体验上，这套系统和模型的“深度思考”很像——都不立刻答，而是先拆解、查资料、再综合。但本质上完全不同。这是我做了一年后觉得最值钱的一个认知：

模型内部的“深度思考”发生在单次推理内部 · 过程不可控 · 不能暂停恢复 · 不能定向重试 · 工具临时决定 · 中间证据易丢失 · 成本不可控 · 一个模型全干完

工程化的深度研究系统拆成多个显式节点 · 任务依赖和状态可控 · 可持久化、能恢复 · 只重试失败的子任务 · Router 按任务匹配 Agent · 证据可存可溯 · 任务数/轮数/预算可限 · 多个专业 Agent 分工

一句话：它实现的是“深度思考”的效果，但用的是 Planner + 任务 DAG + 专业 Agent + Reflector + Synthesizer，把深度思考变成了可执行、可追踪、可重试、可审计的工程流程。

七、它是有状态的研究系统，不是无状态问答

普通问答是问题 → 检索 → LLM → 答案，答完就忘。这套系统有一整套数据对象：

研究会话 · 问题定义 · 计划 · 任务 · 任务依赖 · 执行结果 · 发现 · 证据 · 冲突 · 反思 · 最终报告

这意味着，哪怕 LLM 调用早就结束了，系统仍然知道：用户最初问的是什么、被拆成了哪些任务、哪个 Agent 执行的、每个结论靠什么证据支撑、哪些信息缺失、为什么临时加了任务、最终答案用了哪些中间结果。

这才是研究，不是聊天。前者留下了一整套可追溯的中间产物，后者只留下一句回答。

八、怎么落地：千万别一次重写

最大的坑，是想一上来就把整条链路全做出来。正确做法是分阶段：

第一版 · 最小闭环Planner 拆 3-7 个任务 → 老路由按任务选 Agent → 无依赖任务并行 → 统一收集 → Synthesizer 汇总。先不做 Reflector。

第二版 · 协议化Pydantic 统一任务/结果协议 + 任务依赖 + 并行调度 + 失败重试 + 证据 ID 与引用血缘。

第三版 · 反思与验收加 Reflector（最多一次局部重规划）+ Evaluator + 冲突核验 + 质量和成本监控。

技术栈就这一套：LangGraph（流程、状态、并行、重试、持久化）+ Pydantic（强类型协议）+ PostgreSQL（任务状态和证据）+ 对象存储（文档）+ LangSmith（调用链追踪）。

做 AI Agent，“接 API”是最简单的一步。真正难的，是把“深度思考”变成可执行、可追踪、可重试、可审计的工程流程。

模型负责聪明；你负责让这份聪明——可控、可信、可交付。这才是 AI 产品和“套壳聊天框”的分水岭，也是我们这种想借 AI 重新拿回主动权的程序员，真正要补的硬功夫。
